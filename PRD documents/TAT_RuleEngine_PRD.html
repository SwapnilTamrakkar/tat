01

Product Overview

Healthcare prior authorization workflows are governed by regulatory and contractual Turnaround Time (TAT) requirements. Today, configuring these rules requires developer involvement every time a payer, service type, or regulatory deadline changes — creating bottlenecks and compliance risk.

The TAT Rule Engine Builder is a web application that empowers business users (operations analysts, compliance officers, configuration managers) to independently build, edit, clone, and test TAT rules through a visual interface — with zero developer involvement post-deployment.

Problem Statement

Core Problem
TAT rules are complex (multi-criteria matching + multi-clock behaviors + chaining + exclusion calendars) yet must change frequently to meet regulatory and contractual obligations. The current code-driven approach is too slow, too error-prone, and too expensive for the pace of business change.

Goals

  • Autonomy: Business users configure rules without engineering tickets
  • Correctness: Built-in validation and a simulation sandbox prevent errors from reaching production
  • Auditability: Every rule change is versioned and traceable
  • Scalability: Supports multi-tenant deployments with tenant-specific schedules, holidays, and rules
  • Discoverability: Rules are searchable, filterable, and easy to understand at a glance

User Personas

👩‍💼 Configuration Analyst

Creates and manages TAT rules day-to-day. Familiar with clinical terminology and payer contract terms. Not technical. Needs guided, validated forms with smart defaults.

🛡️ Compliance Officer

Audits rules for regulatory accuracy. Needs rule history, diff views, and the ability to compare rules across tenants. Read access + approval workflows.

🔧 Tenant Admin

Manages tenant-level settings: work schedules, holiday calendars, application registration. Owns the configuration namespace for their organization.

🧪 QA / Tester

Validates rules using the simulation sandbox before they go live. Needs to build test scenarios with mock cases and verify clock behavior output.

Key Structural Insights

Clock Chaining Pattern (Critical UI Feature)
The pause event on the Primary clock is the exact same event that starts the Secondary clock. The stop event on the Secondary clock is also the resume event on the Primary. The UI must represent this as a linked pair — editing the handoff event once auto-propagates to both clocks, ensuring they never drift out of sync.
Pattern Types Drive Exclusion Locking
Two patterns observed: Calendar Days (7-day standard, respects exclusions) and Flat Hours (72-hour expedited, runs continuously). When pattern = Flat Hours, holiday/weekend exclusion fields must be locked/disabled. This is a hard business rule derived directly from the data.
Incomplete Data in Row 4 (OP Physician Expedited)
The 4th rule has a Secondary clock duration (10 Days) but no alert thresholds set. The UI must handle partial secondary clock configurations gracefully — saving the duration without requiring alerts, but surfacing a warning that alerts are unset.

Edge Cases & UX Guardrails

⚠️ Overlapping Rules

Two rules with identical matching criteria cause ambiguous clock selection. Live conflict detection on Step 1 prevents activation. Shows conflicting rule name and offers to deactivate it first.

🔗 Chain Desync

If Secondary start event manually differs from Primary pause event, a persistent warning banner shows. User must explicitly acknowledge before proceeding. Highlighted in simulation output.

⏰ Flat Hours + Exclusions Conflict

If user switches to Flat Hours after setting exclusions, a prompt warns: "Switching to Flat Hours will clear exclusion settings. Continue?" Prevents silent data loss.

📅 Holiday on Start Day

Case submitted on a holiday with Calendar Days + holiday exclusion + Next Day start = clock starts next non-holiday day. Simulation engine must model this accurately; the timeline preview must show it clearly.

🚨 Threshold Order Violation

Warning offset must be greater than Attention offset which must be greater than 0. Real-time inline error. Cannot save if violated. Error message: "Warning threshold must be farther from deadline than Attention."

🕳️ No Matching Rule Gap

Case with no matching rule: Case Debugger surfaces this as "No TAT rule matched this case. Consider adding a rule for [criteria summary]." Prevents silent compliance gaps in production.

🔄 Editing Active Rule

Editing an Active rule creates a Draft copy. Active version stays live until draft is explicitly activated. Persistent banner in editor: "The active version is still applied to new cases until you activate this draft."

📋 Pause Without Secondary

Primary clock has a pause event but no Secondary configured. Warning: "Primary clock has a pause event but pend time is not being tracked. Enable Secondary clock?" User can dismiss and proceed, but the warning persists until resolved.

🌐 Missing Holiday Calendar

Rule references "Client Holidays" but no Client calendar is configured. Warning at save: "No Client Holiday Calendar configured for this tenant. Clock will behave as if no holidays are excluded." Link to calendar setup.

📁 CSV Import Errors

Invalid date format, duplicate entries, unrecognized categories — shown as highlighted rows in import preview. Import blocked if errors exceed 10% of rows. Downloadable error report with row-level detail.

🔁 Circular Event Reference

Start and Stop events cannot be identical condition sets — this creates a paradox. Real-time validation prevents save: "Start and Stop events cannot have identical conditions."

🗑️ Deleted Reference Data

If a Service Type referenced by an active rule is deleted from the registry, all affected rules auto-move to Draft with an alert: "Rule references a deleted service type. Review required."

Open Questions for Stakeholder Review

  1. Should rules have explicit priority/ordering when a case matches multiple rules? The current Excel data implies one rule per unique case profile — needs validation.
  2. Can the same Service Type appear in multiple active rules with different Request Types? The conflict detection algorithm needs a precise definition of "overlap" — AND-based or OR-based matching semantics.
  3. Is the Secondary clock always a "pend" clock, or can it represent other sub-clock types (e.g., appeal clocks, review clocks)? If the latter, a "Clock Purpose" label field is needed.
  4. Does the clock calculation engine need to be exposed via API for real-time case evaluation in external systems, or is this purely a configuration tool that outputs rules consumed by a separate runtime?
  5. What is the maximum number of concurrent clocks per case? Current Excel shows 2 (Primary + Secondary). Are there regulatory use cases requiring 3 or more?
